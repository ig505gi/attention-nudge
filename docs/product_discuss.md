### 1. 插件的 UI/UX 设计
专注力插件的最高设计原则是**“克制”**。它本身绝不能成为新的注意力黑洞。

- **平时状态（隐形）：** 插件在后台默默运行，只在浏览器右上角保留一个小 Icon。点击后弹出一个极简面板，让用户输入“当前专注目标”（例如：正在写代码 / 正在查阅文献），并提供一个一键开启“专注模式”的开关。
- **干预状态（渐进式阻断）：**
  - **轻度提醒：** 网页顶部出现一个细小的 Banner（“当前页面似乎与你的目标偏离，是否需要关掉？”）。
  - **中度提醒：** 弹出一个覆盖屏幕 1/3 的模态框，LLM 给出一段带有同理心的话。
  - **重度阻断：** 屏幕变灰，需要用户手动输入一段文字（比如“我确实需要看这个页面因为…”）才能解锁。增加犯错的“摩擦力”。
- **复盘状态（数据看板）：** 提供当天的“注意力分布图”。用清晰的饼图或时间轴展示时间去向（如：生产力 70%，无意识摸鱼 30%），以及 LLM 生成的每日专注力点评。

### 2. 技术与隐私：记录和监控是否好实现？
**技术实现上确实不难，但细节决定成败：**

- **浏览记录与停留时间：** 极其容易。通过 Chrome Extension API 的 `chrome.tabs` 和 `chrome.webNavigation` 可以轻易获取 URL 和 Title。
- **精准时间统计（难点）：** 仅仅记录打开时间是不够的。你需要结合 `chrome.idle` API 来判断用户是否真的在看这个网页（鼠标是否移动、键盘是否有输入、标签页是否在当前激活状态）。如果用户切到了后台或者去上了个厕所，这段时间不能算作该网页的停留时间。
- **隐私规范（核心红线）：** * **致命风险：** 如果你把用户的网页 DOM（即使是摘要）发送到云端的 LLM API（如 OpenAI），这就构成了实质性的隐私数据出境。如果不明确告知，上架 Chrome Web Store 会被直接拒绝，也极容易引发用户信任危机。
  - **破局方案：** 1.  **脱敏处理：** 提取网页文本后，先在本地跑一个轻量级的正则或 NLP 脚本，剔除手机号、邮箱、密码框内容等敏感信息，再发送给 LLM。
2.  **支持本地大模型：** 这是目前极客圈非常看重的特性。你可以让插件支持配置本地运行的大模型 API（例如接通本地机器上的 Ollama 接口）。这样数据完全不出本地，彻底解决隐私焦虑。

### 3. LLM 的提醒：系统提示词（System Prompt）需要的理论支撑
要让 LLM 成为一个优秀的“专注力教练”，而不是一个只会说“别看了快去工作”的死板机器，你的 Prompt 需要注入以下行为心理学理论：

- **福格行为模型 (Fogg Behavior Model, B=MAP)：**
  - *理论：* 行为发生 = 动机 (Motivation) × 能力 (Ability) × 提示 (Prompt)。
  - *Prompt 应用：* 当 LLM 发现用户偏离目标时，它的回复应该旨在**降低用户继续浏览的动机**，并**提供回到正轨的能力（明确的下一步）**。
  - *示例语料：* “你已经在这个娱乐论坛看了 15 分钟了。它确实很有趣（共情），但你现在的首要任务是完成那段逻辑代码（降低摸鱼动机）。要不我们现在切回 IDE，先写出前三行测试用例？（提供简单的下一步行动）”
- **认知行为疗法 (CBT) 中的“模式中断 (Pattern Interrupt)”：**
  - *理论：* 人在无意识滑手机（Doomscrolling）时是处于一种自动驾驶的惯性状态。干预的目的不是粗暴打断，而是唤醒用户的元认知（让他们意识到自己在干嘛）。
  - *Prompt 应用：* 采用提问式而非命令式。
  - *示例语料：* “系统检测到你正在浏览 YouTube 上的游戏视频，但你的目标是‘学习架构设计’。你是在找相关的学习资料，还是大脑需要休息了？”
- **架构设计建议：多智能体 (Multi-Agent) 协同**
  - 在设计大模型干预机制时，可以采用类似 OpenClaw 这样的多智能体架构思路：
    - **Agent 1 (快速分类器)：** 每次切换网页时触发，负责判断“当前网页内容”与“用户设定的目标”是否相关。这个操作极其高频，应该用响应最快、成本最低的小模型来完成，只输出 `true/false`。
    - **Agent 2 (干预话术生成器)：** 只有当 Agent 1 输出 `false` 时才触发。它接收历史记录、停留时间等上下文，运用上述心理学理论，生成一句高度个性化的提醒文案。

### 1. 核心差异：主动触发 vs. 后台静默监控

- **沉浸式翻译（主动请求）：** 用户是**主动**点击“翻译”按钮，或者明确配置了“遇到英文网站自动翻译”的。在用户的心理预期中：“我需要翻译这个网页，所以我知道并且同意插件读取网页内容”。这符合隐私规范中的**明确授权（Explicit Consent）**。
- **专注力插件（被动监听）：** 你的插件需求是“监控用户的浏览记录”。如果它在后台**静默**读取用户打开的每一个网页（包括网银、公司内网、私人邮件），这对用户来说是不可预期的“全天候监听”。这是浏览器扩展审核的绝对高危区，极易被判定为恶意软件。

### 2. 架构巧思：BYOK (自带密钥) 与直连请求

- 当沉浸式翻译使用 OpenAI 或 Claude 等大模型时，它主推的是让用户填入**自己的 API Key**（BYOK - Bring Your Own Key）。
- 它的网络请求是从用户的浏览器**直接**发往大模型厂商的 API 接口的。
- **关键点：** 用户的网页文本**根本不经过沉浸式翻译开发者的服务器**。开发者不存储、不中转、不分析这些数据，完美避开了“收集/窃取用户隐私”的指控，将数据协议的责任转移到了用户和大模型厂商之间。

### 3. 选择性提取（被动的脱敏）
虽然没有刻意脱敏，但翻译插件在抓取内容时，针对的是 `<p>`, `<h1>`, `<span>` 等常规排版节点，它会天然过滤掉 `<input type="password">`、隐藏的表单等高危输入框，这也降低了泄露核心隐私的风险。

#### 有什么现成的方案可以直接拿来用？
对于 Chrome 插件来说，越轻量越好。你完全不需要复杂的第三方对接系统，以下是三个梯度的落地方案：
**方案 A：原生 Fetch API（最推荐，零依赖，最干净）**
不要用任何复杂的库，直接用 JavaScript 原生的 `fetch` 就能搞定。你只需要在插件的设置页留两个输入框：`选择模型 (API URL)` 和 `输入 API Key`。

```javascript
// 插件 Background Script 中的核心代码逻辑
async function checkAttention(pageSummary, userApiKey, apiUrl) {
  const response = await fetch(apiUrl, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${userApiKey}` // 把用户填写的 Key 塞这里
    },
    body: JSON.stringify({
      model: "deepseek-chat", // 或者 gpt-3.5-turbo
      messages: [
        { role: "system", content: "你是一个专注力教练，判断以下网页内容是否与编程工作相关，只回答 true 或 false。" },
        { role: "user", content: pageSummary }
      ]
    })
  });
  const data = await response.json();
  return data.choices[0].message.content;
}

```
**方案 B：使用官方 SDK (如 OpenAI Node.js SDK)**
如果你觉得手写 Fetch 拼接参数麻烦，你可以直接在插件里引入大模型官方的 NPM 包。

- *注意点：* 默认情况下，OpenAI 的 SDK 会阻止在浏览器环境中运行（怕开发者把自己的 Key 泄露在前端）。既然你是做 BYOK，用的是用户的 Key，你需要在初始化时加上 `dangerouslyAllowBrowser: true` 允许在前端运行。
**方案 C：支持本地大模型（Ollama）—— 终极隐私杀手锏**
既然你是针对极客和程序员群体（他们大多懂一些代码或 Linux），强烈建议你在设置里加一个**“直连本地 Ollama”**的选项。

- 用户在自己的电脑上运行一个几十亿参数的小模型（如 `qwen:0.5b` 或 `llama3:8b`）。
- 你的插件直接向 `http://localhost:11434/api/generate` 发送请求。
- **优势：** 连 API Key 都不需要填

# 基于 LLM 上下文理解的浏览器专注力辅助插件 —— 竞品调研报告
## 一、产品定位与市场背景
在浏览器专注力工具领域，现有产品大致可分为三条技术路径：
传统域名/URL 硬阻断型：以 StayFocusd、LeechBlock NG 为代表，基于用户预设的黑/白名单和时间限额进行网站访问控制。StayFocusd 允许用户为不同网站组设置时间额度（如社交媒体每天 15 分钟），超时后直接屏蔽；LeechBlock NG 则支持最多 30 组网站的分时段屏蔽规则，还提供“冷却等待”功能——用户若需临时访问被屏蔽网站用于研究，需等待冷却时间后方可浏览。这类工具的核心局限在于无法区分同一域名下的生产性使用与娱乐性使用（例如 YouTube 上的技术教程 vs. 娱乐视频），且干预方式为“全或无”的刚性阻断。
事后时间追踪型：以 RescueTime、Toggl Track 为代表，自动记录用户在标签页/网站上的停留时间，生成统计报表和生产力评分。RescueTime 具备空闲状态检测能力——当键盘和鼠标未触及两分钟或更长时间时，计时会自动暂停；Toggl 则提供一键计时和空闲通知功能，支持按项目/任务维度追踪时间。这类工具的核心局限在于仅做“事后记录”，缺乏实时的意图感知和主动干预能力，无法在用户陷入“自动驾驶”状态的当下将其唤醒。
新兴 AI 内容感知型（与你的产品最接近）：利用 LLM 或规则引擎实时分析网页内容，判断与用户当前任务的关联度，并触发柔性干预。这是本调研的重点分析对象。

2. **深度阅读长文本：**

综上，你可以把这些特征组合成一个状态机。一个网页在你的插件里，可以被标记为三种状态之一：

  2. **有效停留触发：** 如之前讨论，用户在不匹配页面活跃超过一定时长 。
- **多层级决策（Hiearchical Reasoning）：**
